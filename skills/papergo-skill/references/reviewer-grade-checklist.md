# Reviewer-Grade Reproduction Checklist

Use this before claiming a reproduction is complete or when auditing an existing reproduction package.

## Blocking Checks

- Paper datasets: every source-stated dataset is downloaded, locally present, or recorded as missing/gated/too-large with concrete acquisition advice.
- Split protocol: train/validation/test or cross-validation setup matches the paper, or the mismatch is labeled as partial.
- Main workflow: every paper main-method step has an implementation reference, command, and output evidence.
- Missing author code: every reconstructed symbol has paper evidence, caller evidence, confidence, and a small replaceable implementation surface.
- No unmarked deviation: main reproduction code does not include convenience changes, alternative sampling rules, new metrics, or synthetic substitutes without explicit labels.
- Metrics: metric definitions match the paper and include denominators/sample counts where possible.
- Outputs: result files are generated by runnable commands, not copied manually from prose.
- README: commands, paths, status label, and limitations match the actual package.

## Non-Blocking But Required Caveats

- Dependencies are not pinned.
- Hardware/runtime is absent.
- Only one seed or one fold is available.
- Standard deviations or confidence intervals are absent.
- Only synthetic mechanism checks were run.
- Some paper baselines are omitted because the user requested main-method-only reproduction.

## Code Review Questions

- Does the implementation use paper formulas and algorithm order rather than an easier adjacent algorithm?
- Are random seeds applied to all stochastic components?
- Are training-only transformations fitted only on training data?
- Is there any test-set leakage through oversampling, scaling, feature selection, or probability estimation?
- Can a human replace uncertain reconstructed helpers without rewriting the full method?
- Do comments distinguish paper-stated behavior from inferred behavior?

## Result Review Questions

- Are all compared methods run on the same dataset, split, preprocessing, classifier, and metric?
- Is the claimed best method best on the paper's primary metric, or only on a chosen secondary metric?
- Does the report separate paper-workflow results from mechanism checks?
- Are failures, missing datasets, and blocked assumptions stated before interpretive conclusions?

## Status Decision

- `ready`: paper datasets, main workflow, metrics, commands, and outputs are covered.
- `partial`: useful implementation or evidence exists, but at least one paper dataset, workflow step, or metric remains incomplete.
- `blocked`: a required dataset, credential, paper detail, or missing author-code component cannot be resolved.
- `mechanism-check`: synthetic or toy experiments validate behavior but do not reproduce paper results.
- `unverified`: outputs exist, but provenance, split, or metric protocol cannot be confirmed.
